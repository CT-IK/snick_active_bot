# Справочник REST API

Базовый префикс — `/api`. Формат — JSON. Все endpoints, кроме входа,
требуют заголовок `Authorization: Bearer <token>`.

Роутеры: [api/auth.py](../api/auth.py), [api/users.py](../api/users.py),
[api/tasks.py](../api/tasks.py), [api/workgroups.py](../api/workgroups.py).
Интерактивная документация FastAPI доступна на `/docs` при запущенном сервере.

Сокращения ролей: **PM** — проектник, **MO** — главный организатор,
**RESP** — ответственный, **WORKER** — работник.

## Аутентификация — `/api/auth`

| Метод | Путь | Доступ | Описание |
|-------|------|--------|----------|
| POST | `/api/auth/login` | публичный | Вход по логину и паролю |

`POST /api/auth/login` — тело `LoginRequest` (`login`, `password`).
Возвращает `TokenResponse` (`access_token`, `token_type`, `user`).
Роль `WORKER` получает `403` — у работников нет доступа к вебу.

## Пользователи — `/api/users`

| Метод | Путь | Доступ | Описание |
|-------|------|--------|----------|
| GET | `/api/users/me` | любой авторизованный | Текущий пользователь |
| POST | `/api/users/me/test-telegram` | любой авторизованный | Отправить себе тестовое сообщение в Telegram |
| GET | `/api/users/` | PM, MO — все; RESP — свои подчинённые | Список пользователей |
| GET | `/api/users/assignable` | PM — все; MO — все кроме PM | Кого можно назначить в группу |
| GET | `/api/users/{id}` | любой; RESP — только своих подчинённых | Пользователь по ID |
| POST | `/api/users/` | все кроме WORKER | Создать пользователя |
| PUT | `/api/users/{id}` | любой; RESP — только своих подчинённых | Обновить пользователя |
| DELETE | `/api/users/{id}` | только PM | Удалить пользователя |

При создании пользователя с веб-ролью (PM/MO/RESP) логин и пароль
обязательны. Логин уникален. Новые поля — `schemas/user.py`.

## Задачи — `/api/tasks`

| Метод | Путь | Доступ | Описание |
|-------|------|--------|----------|
| GET | `/api/tasks/` | любой авторизованный | Список задач; параметр `?workgroup_id=` фильтрует по группе |
| GET | `/api/tasks/assignable-users` | по роли или `?workgroup_id=` | Кого можно назначить исполнителем |
| GET | `/api/tasks/my` | любой авторизованный | Задачи, назначенные текущему пользователю |
| GET | `/api/tasks/{id}` | любой авторизованный | Задача по ID (со связями) |
| POST | `/api/tasks/` | любой авторизованный | Создать задачу |
| PUT | `/api/tasks/{id}` | любой авторизованный | Обновить задачу |
| DELETE | `/api/tasks/{id}` | создатель задачи или PM | Удалить задачу |
| POST | `/api/tasks/{id}/poll-response` | любой авторизованный | Сохранить ответ на опрос |
| POST | `/api/tasks/{id}/nudge` | любой авторизованный | Принудительно отправить напоминание-опрос |

У задачи несколько исполнителей (`assignee_ids`). Поле `assigned_to_id`
оставлено для обратной совместимости. Ответ на опрос продвигает статус
задачи на следующий этап (`new → in_progress → review → done`). Схемы —
`schemas/task.py`.

## Рабочие группы — `/api/workgroups`

| Метод | Путь | Доступ | Описание |
|-------|------|--------|----------|
| GET | `/api/workgroups/` | PM — все; MO — свои созданные; RESP — где он ответственный | Список групп |
| GET | `/api/workgroups/{id}` | любой авторизованный | Группа по ID |
| POST | `/api/workgroups/` | MO, PM | Создать группу |
| PUT | `/api/workgroups/{id}` | создатель группы или PM | Обновить группу |
| DELETE | `/api/workgroups/{id}` | создатель группы или PM | Удалить группу |

MO не может назначить в группу пользователя с ролью PM (проверка
`_can_assign_user` в [api/workgroups.py](../api/workgroups.py)).

## Замечания по контролю доступа

Документирует фактическое поведение кода (важно знать при доработке):

- `GET /api/tasks/`, `GET /api/tasks/{id}`, `PUT /api/tasks/{id}`,
  `GET /api/workgroups/{id}` — доступны **любому авторизованному**
  пользователю без фильтрации по роли или принадлежности. Это точка для
  усиления авторизации при развитии проекта.
- `submit_poll_response` принимает `user_id` в теле запроса.
